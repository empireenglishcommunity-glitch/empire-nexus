## SC1 — Zero non-Arabic-script output reaches a student

**Status: ✅ Strong evidence, verified in Phase A4.**

- The 200-iteration stress test (`test_a4_6_stress_test_200_reproductions_zero_non_conformant_reach_delivery`,
  `tests/test_nour_guardrails.py`) reproduces the exact original bug
  conditions — a flaky provider returning garbled/foreign-language
  text in 6+ languages, plus degenerate empty/whitespace/digit-only
  text — and confirms **0/200** delivered responses ever fail the
  guardrail. This is a structural guarantee (`guarded_generate()`
  withholds delivery of anything that fails `check_script_conformance()`),
  not a probabilistic one — re-verified as still passing as part of
  this phase's full suite run (see "Test suite" below).
- This bar does not require a live API key to verify, since it tests
  the GATE mechanism itself against synthetic bad input, not real
  model behavior.

  fails the bar — it is evidence this specific run could not test the
  real pipeline at all.
- **Action required before A9**: re-run
  `python3 scripts/run_golden_set_eval.py --set student --output results_student.json`

## SC3 — 0/50 red-team attempts succeed

**Status: ✅ Verified — 50/50 passed (0/50 attacks succeeded).**

- `data/nour_eval/red_team_set.json` (50 prompts across 5 categories —
  direct, role_play, instruction_override, obfuscated_translated,
  social_engineering — A8.3) run against
  `scripts/run_golden_set_eval.py --set redteam`.
- **This bar's evaluation is structural, not LLM-output-dependent** —
  it does not ask the model whether it leaked something (an
  attacker-controlled model is not a trustworthy judge of its own
  compliance); it directly inspects: (1) whether `retrieve()` for a
  `Role.STUDENT` call ever surfaced an owner-only `knowledge_chunks`
  domain, (2) whether the tool schema list offered for `Role.STUDENT`
  ever contained an owner-only tool name, (3) whether an owner-only
  tool was ever actually executed (checked against the real
  `nour_tool_calls` log). Because none of these three checks depend on
  what the LLM itself said, **this result is valid even without a real
  API key** — the boundary being tested is the retrieval/tool-schema
  layer (Phases A0–A3), not the LLM's behavior.
- Real result from this sandbox run: **50/50 passed, 0 leaked_domain,
  0 leaked_tool_schema, 0 executed_owner_tool** across all 50 prompts,
  including every role-play, instruction-override, and social-
  engineering framing attempted.

## What COULD be fully verified in this environment (no API key needed)

- **Shadow mode mechanism itself** (`src/nour/shadow.py`, A8.4/A8.5):
  `enable_self_test()` correctly restricts `nour_aql_core` to the
  owner's own Discord ID via the existing `set_feature_flag()`
  allowlist; `is_shadow_eligible()` correctly returns `False` for
  every non-owner `discord_id`; `run_shadow_check()` correctly
  short-circuits to `None` for a non-eligible caller (costing nothing
  beyond one flag lookup) and correctly persists a side-by-side
  comparison record — including the failure path (new pipeline
  raising an exception is caught and logged with an `error` field,
  never propagated to break the caller). All verified by real unit
  tests (`tests/test_nour_shadow.py`), not just by inspection.
- **Red-team structural boundary (SC3)** — see above, fully verified
  with real numbers.
- **SC1's guardrail mechanism** — fully verified in Phase A4, re-confirmed
  still passing in this phase's full suite run.
- **The evaluation harness's own correctness** — confirmed it loads
  all 3 real JSON sets, calls the real pipeline for every item, scores
  correctly, and writes valid JSON output. What it can't do in this
  sandbox is produce a meaningful SC2/SC4 NUMBER, because that requires
  a real model actually answering the questions.
