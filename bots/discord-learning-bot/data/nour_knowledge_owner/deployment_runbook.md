# دليل النشر والتراجع (Deployment Runbook)

> هذا الملف جزء من قاعدة معرفة نور — مخصَّص للمالك فقط. لا يُعرض هذا المحتوى للطلاب أبدًا (Aql Phase A2، domain: `deployment_runbook`).

## مكان الاستضافة

البوت يعمل على خادم **Hetzner VPS** في المجلد `/opt/empire-english-bot/`، داخل حاويات Docker (docker compose). الوصول عبر:
```bash
ssh root@77.42.43.250
cd /opt/empire-english-bot
```

## النشر الآمن الموصى به: `scripts/deploy.py`

هذا هو الطريقة المُفضَّلة، يُشغَّل **من الخادم نفسه (host)**، لا من داخل الحاوية:
```bash
cd /opt/empire-english-bot
python3 scripts/deploy.py
```

### ماذا يفعل بالتفصيل (بالترتيب):

1. **نسخة احتياطية قبل النشر**: `docker exec ... scripts/backup.py --tag pre-deploy-<git-sha>` — تُسمَّى بمعرِّف الإصدار (git SHA) الحالي.
2. **بناء الصورة الجديدة**: `docker compose build --pull`.
3. **وضع علامة على الصورة بمعرِّف git SHA قبل** أن تصبح `:latest` — هذا ما يجعل `rollback.py` قادرًا على العودة لصورة محدَّدة.
4. **تفعيل وضع الصيانة** (`maintenance_mode=on`) ثم تبديل الحاوية للصورة الجديدة (`docker compose up -d`).
5. **فحص صحة النظام**: `scripts/health_check.py` بعد انتظار ٥ ثوانٍ.
6. **عند النجاح**: حذف الصور القديمة المُعلَّمة (الاحتفاظ بآخر ٥)، إلغاء وضع الصيانة، نشر ملخَّص في قناة `#dev-log`.
7. **عند الفشل**: طباعة أمر التراجع الدقيق والخروج بخطأ — **لا تراجع تلقائي**. يجب على المشغِّل فهم سبب الفشل أوَّلًا.

### النشر اليدوي البسيط (بدون شبكة الأمان أعلاه)
```bash
git pull && docker compose up -d --build
```
مناسب فقط لتغيير روتيني بلا داعٍ لحذر إضافي.

## التراجع عن النشر: `scripts/rollback.py`

```bash
python3 scripts/rollback.py                # التراجع لآخر صورة مُعلَّمة
python3 scripts/rollback.py <git-sha>       # التراجع لصورة محدَّدة
python3 scripts/rollback.py --list          # عرض الصور المتاحة للتراجع
```

**الآلية:** إعادة تسمية الصورة المختارة كـ `:latest`، تشغيل `docker compose up -d`، ثم تشغيل `health_check.py` مرة أخرى للتأكُّد من أن التراجع حلَّ المشكلة فعلًا. إذا استمر الفشل بعد التراجع، فالمشكلة على الأرجح **ليست** في الكود — تحقَّق من قاعدة البيانات، متغيِّرات البيئة، أو واجهات خارجية (Discord API، مزوِّدو الذكاء الاصطناعي).

### تراجع قاعدة البيانات (خطوة منفصلة ومتعمَّدة تمامًا)

استعادة نسخة احتياطية من قاعدة البيانات **لا تحدث تلقائيًّا أبدًا** كجزء من تراجع الكود الروتيني — لأنها عملية مُدمِّرة. الخطوات اليدوية:
```bash
# ١. أوقف البوت أوَّلًا (لا كتابة على القاعدة أثناء الاستعادة):
docker compose stop empire-english-bot

# ٢. اعثر على النسخة الاحتياطية المطلوبة (مُسمَّاة بنمط
#    empire_english_pre-deploy-<sha>_<timestamp>.db):
docker exec empire-english-bot ls -la /app/backups/

# ٣. نسخها فوق قاعدة البيانات الحيَّة (داخل mount الحاوية):
docker exec empire-english-bot cp /app/backups/<اسم-الملف>.db /app/data_persist/empire_english.db

# ٤. إعادة التشغيل:
docker compose up -d empire-english-bot
```

## النسخ الاحتياطي الدوري: `scripts/backup.py`

- يحتفظ بآخر ١٤ نسخة، ويحذف الأقدم.
- يُخزَّن في مجلد `backups/` — وهو Docker volume منفصل (`bot-backups`)، يبقى حتى بعد إعادة بناء الحاوية بالكامل.
- **غير مُجدوَل تلقائيًّا إلا إذا أُضيف cron**:
  ```
  0 3 * * * docker exec empire-english-bot python3 scripts/backup.py
  ```

## أعلام الميزات كأداة تراجع فوري (بدون نشر جديد)

هذه هي الآلية الأهم والأسرع لإيقاف ميزة تُسبِّب مشكلة **فورًا، بدون أي إعادة نشر أو حتى SSH**:

- `flag_registry.py` هو المصدر الوحيد لكل الأعلام (كل مبادرة: Aegis, Bawaba, Nabd, Tatawwur, Nour, Markaz, Wuslah, Masar, Hissar).
- `database.sync_flag_registry()` يُسجِّل تلقائيًّا أي علم جديد عند كل بدء تشغيل، دون التأثير على أعلام تم تعديلها يدويًّا من قبل.
- `database.is_feature_enabled(name, discord_id)` يفشل بأمان (fail closed) — علم غير موجود = معطَّل.
- **التبديل الفوري**: أمر Discord `!flag disable <اسم>` أو أمر Telegram `/flag <اسم> off` — كلاهما يُنفَّذ فورًا بلا أي إعادة نشر أو إعادة تشغيل. هذا ما يجعل "شحنت خطأ لكل الطلاب" تصبح "شحنت خطأ لأحد فورًا يعود صفرًا" — دون لمس الكود الجاري.
- علم `nour_aql_core` بالتحديد هو مسار التراجع الفوري الكامل لمبادرة Aql بأكملها: إيقافه يُعيد النظام فورًا لخطِّ أنابيب `nour_concierge.py` القديم، دون إعادة نشر.

## بعد أي نشر: قائمة تحقُّق سريعة

1. تحقَّق من حالة البوت في Discord (يجب أن يظهر "متصل").
2. نفِّذ `!status` للتأكُّد من عمل كل الخدمات.
3. راقب سجلات الحاوية: `docker compose logs --tail=50`.
4. تحقَّق من قناة `#dev-log` — يجب أن يظهر سجل النشر تلقائيًّا.
5. إذا كانت الميزة الجديدة خلف علم (flag)، لا تُفعِّلها فورًا لكل الطلاب — فعِّلها لنفسك أوَّلًا، ثم لعدد محدود، ثم للجميع (نمط تدريجي مُعتمَد في كل مبادرات هذا المشروع).
